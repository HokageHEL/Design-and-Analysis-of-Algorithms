# Лабораторна робота №1 — Тестування швидкодії алгоритмів сортування

## Що робилось

Метою роботи було реалізувати три класичні алгоритми сортування, виміряти їхню реальну швидкодію на масивах різного розміру (1K, 5K, 10K, 25K) у трьох випадках вхідних даних (відсортований, випадковий, зворотно відсортований) та порівняти отримані результати з теоретичною асимптотичною складністю Big O.

## Як це було зроблено

### 1. Генерація тестових масивів — `Testing/ArrayGenerator.cs`

Реалізовано три методи генерації:

- **`GenerateSortedArray(size)`** (`Testing/ArrayGenerator.cs:15-18`) — створює відсортований масив `[1, 2, …, size]` через `Enumerable.Range(1, size).ToArray()`.
- **`GenerateReverseSortedArray(size)`** (`Testing/ArrayGenerator.cs:10-13`) — той самий діапазон, перевернутий через `.Reverse()`.
- **`GenerateRandomlySortedArray(size)`** (`Testing/ArrayGenerator.cs:20-31`) — застосовано **алгоритм Фішера-Йєйтса**: цикл `for (int i = arr.Length - 1; i >= 1; i--)` (рядок 24) проходить масив з кінця до початку та обмінює поточний елемент із випадково вибраним елементом з діапазону `[0, i]` через `(arr[i], arr[j]) = (arr[j], arr[i])` (рядок 27). Це гарантує рівномірну випадкову перестановку. Для відтворюваності `Random` ініціалізовано фіксованим seed = 42 (`Testing/ArrayGenerator.cs:8`).

### 2. Алгоритми сортування — `Algorithms/`

Усі три алгоритми працюють на копії вхідного масиву (`array.Clone()`), щоб не модифікувати оригінал, та повертають об'єкт `SortingResult` з масивом, кількістю порівнянь та обмінів.

**Insertion Sort** (`Algorithms/InsertionSort.cs:7-28`) — O(n²) у середньому, O(n) на вже відсортованому масиві. На кожній ітерації зовнішнього циклу (рядок 13) береться `key = arr[i]`, далі внутрішній `while (j >= 0 && ++comparisons > 0 && arr[j] > key)` (рядок 18) зсуває назад елементи більші за нього. Лічильник `comparisons` інкрементується вбудовано в умову циклу, а `swaps++` (рядок 21) рахує кожен зсув.

**Quick Sort** (`Algorithms/QuickSort.cs:7-47`) — O(n log n) у середньому, O(n²) у найгіршому випадку. Реалізовано класичну схему Lomuto: метод `Partition` (`Algorithms/QuickSort.cs:28-47`) обирає опорним елементом `arr[high]` (рядок 30), у циклі `for (int j = low; j < high; j++)` (рядок 33) переставляє елементи менші/рівні pivot вліво. Порівняння рахується на рядку 35, обмін кортежем `(arr[i], arr[j]) = (arr[j], arr[i])` — на рядку 39. Рекурсивний виклик відбувається в `QuickSortRecursive` (`Algorithms/QuickSort.cs:18-26`).

**Merge Sort** (`Algorithms/MergeSort.cs:7-56`) — O(n log n) у всіх випадках. Класичне рекурсивне розділення на дві половини в `MergeSortRecursive` (`Algorithms/MergeSort.cs:18-27`) з подальшим злиттям через тимчасові масиви `L` та `R` (`Algorithms/MergeSort.cs:34-35`). Цикл злиття `while (i < n1 && j < n2)` (рядок 42) обирає менший елемент — порівняння рахується на рядку 44.

### 3. Відображення результатів — `Reporting/ConsoleTableRenderer.cs`

Зібрана таблиця у псевдографічних межах (рядки 9-11) з шістьма колонками: назва алгоритму, розмір масиву, тип випадку, час виконання в мс, кількість порівнянь та обмінів. Заголовок таблиці — `Reporting/ConsoleTableRenderer.cs:21`. Основний цикл рендерингу — `Reporting/ConsoleTableRenderer.cs:24-48`, де використано **інтерполяцію рядків з форматуванням**: `{r.ArraySize,10:N0}` (рядок 35) для вирівнювання та групування числа за тисячами, `{r.AlgorithmName,-15}` для лівого вирівнювання тексту в колонці фіксованої ширини. Після кожних трьох рядків (різні випадки одного розміру) додається роздільник `Mid` — умова `(i + 1) % 3 == 0` на рядку 38.

### 4. Заміри — `Testing/PerformanceMeasurer.cs` + `Program.cs`

Час виконання вимірюється через клас `Stopwatch` усередині `PerformanceMeasurer.MeasureAlgorithm`. У `Program.cs:31-36` створюється список з трьох алгоритмів, у циклі `Program.cs:45-58` для кожного прокручуються всі комбінації розмір × випадок, результати збираються у `ComplexityMeasurement` і на рядку 70 передаються рендереру.

## Висновки за результатами

Емпіричні заміри підтверджують теорію:

- Insertion Sort на відсортованому масиві працює лінійно швидко (внутрішній `while` на `Algorithms/InsertionSort.cs:18` одразу виходить), а на випадковому й зворотньо відсортованому час зростає квадратично — на 25K елементів стає на порядки повільнішим за два інших алгоритми.
- Quick Sort показує найкращу практичну швидкодію на випадкових даних, але на вже відсортованих масивах деградує до O(n²) через невдалий вибір опорного елементу `arr[high]` (`Algorithms/QuickSort.cs:30`).
- Merge Sort має стабільний час O(n log n) незалежно від типу вхідних даних, ціною додаткової пам'яті на тимчасові масиви (`Algorithms/MergeSort.cs:34-35`).
