# Лабораторна робота №3 — Розширене червоно-чорне дерево (рейтинг гравців)

## Що робилось

Завданням було реалізувати **розширене (augmented) червоно-чорне дерево** з додатковим полем `Size` у кожному вузлі та збудувати на ньому систему рейтингу гравців League of Legends. Структура повинна підтримувати мільйон гравців, швидкий пошук позиції в рейтингу та оновлення MMR (Match Making Rating) з автоматичним перебалансуванням.

## Як це було зроблено

### 1. Компаратор гравців — `Models/PlayerComparer.cs:5-17`

Реалізовано `IComparer<Player>` з двоступеневим порівнянням:

1. **Первинне** — за MMR за спаданням: вищий MMR = вища позиція в дереві. Для цього на рядку 12 використано `y.MMR.CompareTo(x.MMR)` (поміняні аргументи, щоб отримати спадний порядок).
2. **Tiebreaker** — при однаковому MMR за іменем гравця за зростанням (`Models/PlayerComparer.cs:16`): `string.Compare(x.SummonerName, y.SummonerName, StringComparison.Ordinal)` для детермінованого побайтового порівняння.

Окремо обробляються `null`-випадки (`Models/PlayerComparer.cs:7-9`) для відповідності контракту `IComparer<T>`.

### 2. Розширене червоно-чорне дерево — `DataStructures/AugmentedRedBlackTree.cs`

Класичне червоно-чорне дерево, у якому кожен вузол додатково зберігає **`Size`** — кількість вузлів у його піддереві (поле визначене в `DataStructures/AugmentedRedBlackTree.cs:14`). Це дозволяє за O(log n) знаходити елемент за рангом без явного обходу.

**`GetNodeAtRank(node, rank)`** (`DataStructures/AugmentedRedBlackTree.cs:248-260`) — рекурсивний пошук вузла за позицією у відсортованому порядку. На рядку 252 беремо `leftSize = GetSize(node.Left)`:
- якщо `rank == leftSize + 1` (рядок 254) — знайшли потрібний вузол;
- якщо `rank <= leftSize` (рядок 256) — шукаємо в лівому піддереві;
- інакше (рядок 258) — у правому піддереві з `rank - leftSize - 1` (віднімаємо ліве піддерево + сам вузол).

**`RotateLeft(x)`** (`DataStructures/AugmentedRedBlackTree.cs:435-457`) і **`RotateRight(y)`** (`DataStructures/AugmentedRedBlackTree.cs:459-481`) — стандартні ротації червоно-чорного дерева, але **критично важливо** після ротації перерахувати `Size` для двох вузлів. Це робиться на рядках 455-456 для `RotateLeft` (спочатку нижній `x`, потім верхній `y`) та на рядках 479-480 для `RotateRight`. Без цього `GetNodeAtRank` повертатиме невірні результати після перших же балансувань.

**Допоміжні методи:** `GetSize(node)` (`DataStructures/AugmentedRedBlackTree.cs:275`) повертає `node?.Size ?? 0`, а `UpdateSizesUpward(node)` (`DataStructures/AugmentedRedBlackTree.cs:277-284`) проходить вгору від вузла до кореня, перераховуючи розміри після вставки/видалення.

**Інша інфраструктура дерева:**
- Вставка з `Insert` (`DataStructures/AugmentedRedBlackTree.cs:68-107`) і подальшим виправленням `FixInsert` (`DataStructures/AugmentedRedBlackTree.cs:286-347`) за класичними 3 випадками (uncle red — рекольорування, uncle black + node right child — поворот);
- Видалення з `Delete` → `DeleteNode` (`DataStructures/AugmentedRedBlackTree.cs:170-224`) і `FixDelete` (`DataStructures/AugmentedRedBlackTree.cs:349-433`);
- Інваріант `_root.Color = Black` встановлюється в кінці `FixInsert` на рядку 346.

### 3. Реалізація рейтингу — `Implementations/RBTreeLeaderboard.cs`

Леaderboard поєднує дерево з допоміжним `Dictionary<string, Player> _playerLookup` (`Implementations/RBTreeLeaderboard.cs:6`) для O(1) пошуку гравця за ім'ям.

**`UpdatePlayerMMR(name, newMMR)`** (`Implementations/RBTreeLeaderboard.cs:82-90`) — щоб перемістити гравця в новій позиції, потрібно спочатку **видалити** його зі старим MMR (рядок 87: `_tree.Delete(player)`), потім оновити поле через `player.UpdateMMR(newMMR)` (рядок 88) та **вставити заново** (рядок 89: `_tree.Insert(player)`). Просто змінити MMR на місці не можна — це порушило б інваріант BST.

**`GetRankDistribution()`** (`Implementations/RBTreeLeaderboard.cs:92-102`) — повертає `Dictionary<RankTier, int>` з кількістю гравців у кожному з рангів (Iron, Bronze, Silver, …, Challenger). На рядках 94-96 ініціалізується словник з нулями для всіх рангів через `Enum.GetValues<RankTier>()`, потім у циклі `foreach (var player in _tree.InOrderTraversal())` (рядок 98) інкрементується лічильник по `player.Tier`. Складність — O(n) за один прохід.

**`BulkAddPlayers`** (`Implementations/RBTreeLeaderboard.cs:27-51`) — для початкового завантаження 1 000 000 гравців використовується пакетна вставка через `_tree.BulkInsert(uniquePlayers)` (рядок 43), попередньо відфільтровуючи дублікати по імені через звичайний `HashSet<string>` (рядки 33-41). Метод `BulkInsert` дерева (`DataStructures/AugmentedRedBlackTree.cs:33-48`) сортує елементи за O(n log n) та будує збалансоване дерево через `BuildBalancedTree` (`DataStructures/AugmentedRedBlackTree.cs:50-66`) рекурсивним розбиттям масиву на половини — швидше, ніж n окремих вставок.

## Чому червоно-чорне дерево

Звичайний `Dictionary` дає O(1) на пошук, але не зберігає порядок — щоб дізнатись ранг гравця або топ-10, довелося б щоразу сортувати весь список за O(n log n). Червоно-чорне дерево гарантує висоту O(log n) у найгіршому випадку, тому всі ключові операції — вставка, видалення, пошук, отримання за рангом (`GetNodeAtRank` на `DataStructures/AugmentedRedBlackTree.cs:248`), обчислення рангу (`GetRank` на `DataStructures/AugmentedRedBlackTree.cs:109`) — виконуються за O(log n). На мільйоні гравців це означає ~20 кроків замість мільйона.
